# Commit Guide

본 프로젝트의 커밋 메시지는 코드의 변경 이력을 명확히 하고 협업 효율을 높이기 위해 아래 규칙을 따릅니다.

## 1. 커밋 메시지 구조
```text
<type>(<scope>): <subject>

<body>

<footer>
```

## 2. Type 가이드 (영문)
- `feat`: 새로운 기능 추가
- `fix`: 버그 수정
- `refactor`: 코드 리팩토링 (기능 변화 없음, 구조 개선)
- `docs`: 문서 수정 (README, guide 등)
- `style`: 코드 포맷팅, 세미콜론 누락 등 (로직 변경 없음)
- `test`: 테스트 코드 추가 및 수정
- `chore`: 빌드 업무, 패키지 매니저 설정, 단순 설정 변경 등

## 3. 제목(Subject) 규칙
- **영문 작성**을 원칙으로 하며, 50자 이내로 간결하게 작성합니다.
- 첫 글자는 소문자로 시작하며, 끝에 마침표(.)를 붙이지 않습니다.
- 과거 시제가 아닌 **명령형**을 사용합니다. (예: `add`, `fix`, `update`)

## 4. 본문(Body) 규칙
- **한국어로 작성**하여 변경 이유와 상세 내역을 명확히 전달합니다.
- "무엇을", "왜" 변경했는지를 중심으로 기술합니다.
- 여러 개의 변경 사항이 포함된 경우 불렛 포인트(`-`)를 사용하여 구분합니다.
- 한 줄당 72자 이내로 작성하는 것을 권장합니다.

## 5. 예시 (Example)
```text
refactor(trading): implement bottom-up weighted scoring

- 가중치 기반 Bottom-Up 평가 방식 도입 (20/20/30/30)
- Section 3에 대량 거래량 돌파에 대한 과열 예외 조항 추가
- JSON 출력 형식을 가중치 항목 수(Max 4)에 맞춰 업데이트
```

## 6. 자동 커밋 및 추가 금지 규칙 (AI Assistant 전용)
- 코드 수정 후 `git add` 및 `git commit`은 자동으로 수행하지 않습니다.
- `git add`와 `git commit` 명령어는 오직 사용자가 명시적으로 요청한 경우에만 수행해야 합니다.
